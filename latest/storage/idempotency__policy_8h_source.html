25 /**
26  * Define the interface for the idempotency policy.
27  *
28  * The idempotency policy controls which requests are treated as idempotent and
29  * therefore safe to retry on a transient failure. Non-idempotent operations can
30  * result in data loss. For example, consider `DeleteObject()`, if this
31  * operation is called without pre-conditions retrying this operation may delete
32  * more than one version of an object, which may not have the desired effect.
33  * Even operations that "add" data can result in data loss, consider
34  * `InsertObjectMedia()`, if called without pre-conditions retrying this
35  * operation will insert multiple new versions, possibly deleting old data if
36  * the bucket is configured to keep only N versions of each object.
37  *
38  * Some applications are designed to handle duplicate requests without data
39  * loss, or the library may be used in an environment where the risk of data
40  * loss due to duplicate requests is negligible or zero.
41  *
42  * This policy allows application developers to control the behavior of the
43  * library with respect to retrying non-idempotent operations. Application
44  * developers can configure the library to only retry operations that are known
45  * to be idempotent (that is, they will succeed only once). Application may also
46  * configure the library to retry all operations, regardless of whether the
47  * operations are idempotent or not.
48  */
